# Shared hosting modded runtime and content compiler handoff

## Product decision

Shared hosting targets **modded Minecraft servers**, not vanilla-only hosting.
Users may bring supported modpacks from any Minecraft version, including legacy
Java 8-era packs. A vanilla test container is not an acceptable product or
staging substitute.

The shared node’s Minecraft containers have an internal Docker network and no
outbound Internet access by design. Therefore they must never download Forge,
Fabric, NeoForge, Quilt, Minecraft server jars, mod artifacts, or installers at
start time.

## Scope

This is one cohesive cross-repository package:

```text
C:\Users\ci010-4090\Workspace\xmcl-web-api
C:\Users\ci010-4090\Workspace\xmcl-shared-node-agent
```

Create a deterministic, trusted modpack compilation pipeline and a generic
multi-JRE runtime image contract. Integrate it with the existing:

- validated Modrinth `.mrpack` and CurseForge ZIP import/deployment metadata;
- v2 immutable shared workspace content/config/world blob layout;
- command/lease-bound object-store grants;
- shared-service stop/sync/start lifecycle.

Do **not** weaken node isolation, reintroduce static Object Storage credentials,
allow arbitrary Docker images/commands/environment variables from users, or
download loader artifacts in a running customer container. Do not change
billing or public shared-hosting activation gates except where a new authenticated
shared deployment API is needed.

## Architecture

```text
modpack import + validated manifest
          |
          v
trusted compiler job (separate egress-enabled builder sandbox)
          |
          |-- resolve + hash all loader/server/mod artifacts
          |-- install server loader/runtime
          |-- generate immutable runtime descriptor
          |-- package content layer
          v
service-owned immutable content blob + descriptor
          |
          v
shared node restore -> generic multi-JRE runtime image -> validated launch
```

The compiler is a trusted server-side workload. It may have constrained outbound
network access only to approved artifact origins. It must run outside the
customer Minecraft node/container and receive no Worker, billing, Object
Storage master, or node-credential secret.

## Versioned runtime descriptor

Define a canonical `schemaVersion: 1` runtime descriptor in the immutable
content layer, for example `.xmcl/runtime.json`:

```json
{
  "schemaVersion": 1,
  "minecraftVersion": "1.12.2",
  "javaMajor": 8,
  "loader": {
    "kind": "forge",
    "version": "14.23.5.2860"
  },
  "launch": {
    "kind": "generated-server-launcher",
    "path": ".xmcl/launch.sh",
    "arguments": []
  },
  "contentSha256": "..."
}
```

This descriptor is generated by the compiler, never trusted from an uploaded
archive verbatim. The signed shared workspace manifest must bind the descriptor
digest as part of the immutable content blob.

The generic runtime image accepts only the descriptor and fixed validated
arguments. It must reject:

- unsupported schema/loader/JRE;
- arbitrary executable path, shell expressions, JVM flags, environment values,
  container image names, host paths, URLs, or Docker options;
- a descriptor/content hash mismatch;
- unapproved loader artifact origins.

## Loader and Java support

Initial supported loader enum:

```text
forge
fabric
neoforge
quilt
```

Design the enum to be extensible for future plugin servers without treating
them as mod loader aliases.

Support all Java eras through trusted JREs in the generic runtime image:

| Java major | Primary compatibility |
| --- | --- |
| 8 | legacy Forge/Fabric packs (including 1.12.2-era) |
| 17 | modern 1.17 through 1.20.4-era packs |
| 21 | 1.20.5+ and current packs |

The compiler resolves `javaMajor` from validated loader/Minecraft compatibility
metadata. It must not guess solely from a broad Minecraft-version string where
the loader has a known exception. Unknown mappings fail with an explicit
unsupported-compatibility result; do not silently select a different Java.

## Compiler inputs and sandbox

Reuse existing validated import and resolved source data where possible.
Compiler input includes:

- account/service/deployment identity;
- frozen deployment manifest and manifest hash;
- validated Minecraft version, loader kind/version, Java major;
- resolved mod artifact URLs and SHA-256 hashes;
- validated config/defaultconfig/data files;
- compiler request idempotency key.

The compiler:

1. Creates a clean ephemeral filesystem and non-root sandbox.
2. Downloads only validated loader/server/mod artifacts through allowlisted
   HTTPS origins with bounded redirects, response sizes, timeouts, and SHA-256
   verification.
3. Runs required loader installation in an isolated builder container/process
   with no secrets, no Docker socket, no host mounts outside the workspace,
   resource/PID limits, and no network after artifact acquisition.
4. Writes the generated launch metadata and descriptor, normalizes ownership
   and paths, and rejects symlinks/devices/FIFOs/setuid files.
5. Packages `runtime/`, `mods/`, `kubejs/`, scripts, resource packs, server
   configuration, and descriptor into deterministic immutable content.
6. Uploads the content object with compiler-specific, deployment-bound
   pre-signed PUT grants. It must not use shared-node command grants or static
   Object Storage credentials.
7. Publishes a durable deployment-to-content record only after every object
   hash verifies.

Compilation must be idempotent: retries for the same frozen deployment produce
or reuse the same content digest; they never overwrite a published object.

## Generic multi-JRE runtime image

Build and publish a trusted image owned by XMCL, e.g.

```text
ghcr.io/voxelum/xmcl-shared-minecraft-runtime:<immutable digest>
```

It contains pinned Java 8, 17, and 21 runtimes plus a small non-root launcher.
It must:

- run as UID/GID 1000 and use `/data` as its only writable application state;
- use a Docker health check that validates the Minecraft server is reachable
  locally, not merely that a shell process exists;
- read and validate `.xmcl/runtime.json`;
- select only the declared bundled Java executable;
- execute only the compiler-generated launcher with validated fixed arguments;
- set `eula=true` only after the hosting product’s terms/EULA acceptance policy
  is satisfied and recorded server-side;
- have no package manager, shell-based user hooks, Docker socket, privileged
  capability, or runtime artifact download path;
- work with the existing read-only root filesystem, cap drop, PID/memory/CPU
  limits, private network, `/data` mount, and assigned port binding.

Do not use `itzg/minecraft-server` as the shared production runtime: it
downloads/installs server dependencies dynamically and is not sufficient for
the network-isolated arbitrary-version modpack contract.

## Shared deployment lifecycle

Add a shared-service deployment projection/API separate from legacy dedicated
`/v1/servers/:serverId/modpack-*` routes:

```text
POST /v1/shared-hosting/services/:serviceId/modpack-imports
POST /v1/shared-hosting/services/:serviceId/modpack-deployments
GET  /v1/shared-hosting/services/:serviceId/modpack-deployments
```

Exact URL shape may differ, but ownership and idempotency are mandatory.

Rules:

1. A deployment targets a service owned by the authenticated account and its
   active shared subscription.
2. Applying a new content layer to a running service first performs safe
   stop-and-sync. It never mutates a running container’s files.
3. The compiler publishes the new content descriptor/digest.
4. A subsequent start restores the previous world/config revision plus the
   selected new immutable content layer. The scheduler may place it on another
   node.
5. Failed compilation leaves the current published content and last complete
   world revision untouched.
6. Rollback selects a prior published deployment/content digest and performs
   the same stopped transition; it does not roll back the user world unless an
   explicit world restore is requested separately.

Refactor/reuse existing modpack validation and frozen manifest code where
possible, but do not couple shared deployment to dedicated-worker staging or
server resource IDs.

## Object Storage grants

Add a compiler-specific server-side grant authority that is bound to:

```text
accountId
serviceId
deploymentId
frozen deployment manifest hash
exact content key
method
expiration
```

Compiler grants may only PUT its service-owned immutable content key and GET
the precise validated input objects it needs. They cannot List/Delete, access
world revisions, or impersonate a node. Shared-node restore grants permit GET
only after the content record is published and selected by the service
deployment.

## Tests

Web API:

1. Legacy and modern compatibility resolve Java 8/17/21 correctly; unknown
   combinations fail explicitly.
2. A compiler request has immutable input identity, safe artifact allowlisting,
   hash validation, bounded retries, and no cross-account/service key access.
3. A failed compiler run cannot change current selected content or world
   revision.
4. Shared deployment of a running service requires stop/sync before a new
   descriptor is selectable.
5. Rollback selects prior content without rewriting world data.
6. Compiler grants and node grants cannot be substituted for each other.

Runtime/agent:

1. Descriptor validation rejects path traversal, arbitrary command/flags,
   unknown loader/JRE, digest mismatch, and untrusted launcher.
2. Java 8, 17, and 21 descriptor fixtures select only their bundled JRE.
3. A Forge, Fabric, NeoForge, and Quilt prebuilt fixture starts without
   outbound dependency download.
4. Existing Docker hardening stays effective.

Run targeted Deno tests/check, `gofmt`, `go test ./...`, `go vet ./...`, and
image build/health-check tests.

## Acceptance staging

Before public shared-hosting activation, demonstrate at least:

```text
legacy Java 8 Forge pack
modern Java 17 Fabric pack
current Java 21 NeoForge or Fabric pack
```

For each:

```text
import -> compile -> immutable content publish -> service start -> health ->
external Minecraft connect -> stop/sync -> restart on another node -> same
modded world/config/content
```

No customer runtime container may access the Internet or use storage
credentials during these tests.
